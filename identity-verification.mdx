---
title: "Identity verification"
description: "Prevent spoofed identify calls by signing user UIDs with a backend secret."
---

By default, any client-side code could call `harpy.identify()` with an arbitrary user ID. Identity verification closes that gap — your backend generates a cryptographic signature for each user, and Harpy rejects any identify call that doesn't include a valid one.

<Note>
  This is recommended for production deployments where data integrity matters. Once enabled, unverified identify calls are rejected.
</Note>

---

## How it works

1. Your backend generates a **SHA-256 HMAC** of the user's UID using a secret key provided by Harpy
2. The hash is passed to `harpy.identify()` alongside the user's other attributes
3. Harpy validates the hash server-side — if it doesn't match, the call is rejected

The secret key never leaves your backend. The hash is computed per-user at runtime.

---

## 1. Generate the hash on your backend

Compute the HMAC using your Harpy secret key and the user's UID. Examples across common backend languages:

<Tabs>
  <Tab title="Node.js">
    ```javascript
    const crypto = require('crypto');

    const uid_hash = crypto
      .createHmac('sha256', process.env.HARPY_SECRET_KEY)
      .update(user.id)
      .digest('hex');
    ```
  </Tab>
  <Tab title="Ruby">
    ```ruby
    require 'openssl'

    uid_hash = OpenSSL::HMAC.hexdigest(
      'SHA256',
      ENV['HARPY_SECRET_KEY'],
      user.id.to_s
    )
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import hmac, hashlib, os

    uid_hash = hmac.new(
      os.environ['HARPY_SECRET_KEY'].encode(),
      user.id.encode(),
      hashlib.sha256
    ).hexdigest()
    ```
  </Tab>
  <Tab title="Go">
    ```go
    import (
      "crypto/hmac"
      "crypto/sha256"
      "encoding/hex"
    )

    mac := hmac.New(sha256.New, []byte(os.Getenv("HARPY_SECRET_KEY")))
    mac.Write([]byte(userID))
    uidHash := hex.EncodeToString(mac.Sum(nil))
    ```
  </Tab>
</Tabs>

Expose the hash to your frontend — for example, by embedding it in your server-rendered HTML or returning it from a `/session` API endpoint.

---

## 2. Pass the hash to `harpy.identify()`

Include `uid_hash` alongside your other user attributes:

```javascript
harpy.identify(USER.ID_IN_DB, {
  email:    USER.EMAIL,
  name:     USER.NAME,
  uid_hash: USER.UID_HASH,  // generated by your backend
  // ... other attributes
});
```

---

## 3. Store your secret key securely

<Warning>
  Never expose your Harpy secret key in client-side code or commit it to version control. Store it as an environment variable and access it server-side only.
</Warning>

```bash
# .env (server-side only)
HARPY_SECRET_KEY=your_secret_key_here
```

---

## Troubleshooting

**Users aren't being identified after enabling verification**
The most likely cause is a missing or incorrect `uid_hash`. Confirm the hash is being generated with the correct secret key and that the same UID string is used in both the hash and the `harpy.identify()` call (watch for type mismatches, e.g. integer vs string).

**Hash is valid but calls are still rejected**
Ensure the UID passed to `harpy.identify()` exactly matches the value used to generate the hash — including type, casing, and any padding.
